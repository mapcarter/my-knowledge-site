# 左侧栏层级关系与维护指南

本文说明本知识库网站左侧目录栏的生成规则、当前层级结构，以及新增、移动、重命名和删除目录或文章时的操作方法。

本文位于仓库根目录，不属于 `docs/` 发布目录，因此不会显示在网站左侧栏中。

## 一、基本规则

本网站使用 MkDocs Material，当前 `mkdocs.yml` **没有配置 `nav`**。因此左侧栏不是手工逐项写出来的，而是 MkDocs 根据 `docs/` 下的目录和 Markdown 文件自动生成的。

必须先区分四种对象：

| 对象 | 示例 | 左侧栏中的作用 |
|---|---|---|
| 站点名称 | `我的科研知识库` | 顶部站点名称，通常不是页面链接 |
| 目录 | `docs/ai/` | 导航分组，用于表示一级目录 |
| `index.md` | `docs/ai/AI与大模型/index.md` | 对应目录的首页或分类说明页，是一个可点击页面 |
| 普通 Markdown 文件 | `prompt-engineering.md` | 一篇可点击的文章页面 |

本项目对新增内容采用以下内容组织原则：

```text
docs/
└── 一级目录/
    └── 二级目录/
        ├── index.md       # 二级目录首页或分类说明页
        └── 文章.md         # 具体文章
```

其中，AI 分类已经按这一规则整理。`embedded/`、`papers/` 和 `traffic-prediction/` 中仍有历史示例文章直接位于一级目录下；后续新增内容不要继续采用这种旧结构，已有内容可在需要时逐步迁移。

也就是说：

- `docs/` 是网站公开内容的根目录。
- 一级目录只负责分类，不直接放文章。
- 一级目录下面放二级目录。
- 文章只能放在二级目录下面。
- 二级目录可以有一个 `index.md`，作为该二级目录的首页或分类介绍页。
- 有无图片的文章都直接使用二级目录下的 `.md` 文件；有图片时，将图片放入该二级目录的 `images/`，并使用文章名前缀避免重名。

### 2.1 AI 部分的准确解释

当前 AI 部分可以简化表示为：

```text
Ai                                  # 一级目录：docs/ai/
├── AI工具                            # 二级目录：docs/ai/AI工具/
│   └── AI 工具                        # 二级目录首页：docs/ai/AI工具/index.md
└── AI 与大模型                       # 二级目录：docs/ai/AI与大模型/
    ├── AI 与大模型                   # 二级目录首页：docs/ai/AI与大模型/index.md
    ├── 大模型技术综述                # 文章：llm-survey.md
    └── Prompt 工程入门                # 文章：prompt-engineering.md
```

这里有两个容易混淆的名称：

- `AI与大模型` 是文件夹名称，表示二级目录。
- `AI 与大模型` 是该目录中 `index.md` 的页面标题，表示这个二级目录的首页。
- `AI工具` 是文件夹名称，表示二级目录。
- `AI 工具` 是该目录中 `index.md` 的页面标题，表示这个二级目录的首页。

因此，目录名称和 `index.md` 的标题可能显示为两行。目录行用于展开和收起，`index.md` 行用于打开该目录的首页。两者不是同一个对象。

## 三、文件名、页面标题与显示文字

普通 Markdown 文件通常按照以下优先关系显示标题：

1. 如果文件有 frontmatter 中的 `title`，通常使用 `title`。
2. 否则使用文档中的第一个一级标题，例如 `# Prompt 工程入门`。
3. 如果没有明确标题，则可能回退为文件名。

示例：

```markdown
---
title: Prompt 工程入门
tags: [ai, prompt, llm]
---

# Prompt 工程入门
```

文件名是：

```text
prompt-engineering.md
```

但左侧栏显示的是：

```text
Prompt 工程入门
```

因此：

- 修改文章在左侧栏显示的名称，优先修改 frontmatter 的 `title`。
- 修改目录在左侧栏显示的名称，修改目录名。
- 修改 `index.md` 的页面名称，修改该文件的 `title` 或一级标题。
- 修改文件名可能会改变页面 URL，应同时检查所有引用该文件的链接。

## 四、新增二级目录

例如，在 AI 一级目录下新增“AI编程”二级目录：

### 4.1 创建目录首页

```text
docs/ai/AI编程/index.md
```

内容示例：

```markdown
---
title: AI 编程
---

# AI 编程

收录 AI 编程工具、开发流程和实践记录。
```

### 4.2 添加文章

文章应放在该二级目录下：

```text
docs/ai/AI编程/coding-agent.md
docs/ai/AI编程/ide-assistant.md
```

每篇文章建议包含 frontmatter：

```markdown
---
title: Coding Agent 使用记录
tags: [ai, 编程, agent]
---

# Coding Agent 使用记录

正文内容。
```

完成后，结构应为：

```text
docs/ai/
└── AI编程/
    ├── index.md
    ├── coding-agent.md
    └── ide-assistant.md
```

同时在 `docs/ai/AI编程/index.md` 中维护内容目录：

```markdown
## 内容目录

- [Coding Agent 使用记录](./coding-agent.md)
- [IDE Assistant 使用记录](./ide-assistant.md)
```

`index.md` 不会自动生成文章列表；目录下每次新增、删除或重命名文章，都要同步更新这份列表。

不应将文章直接放到下面的位置：

```text
docs/ai/coding-agent.md        # 错误：文章直接位于一级目录下
```

## 五、新增文章

### 5.1 纯文字文章

直接在对应二级目录下创建 Markdown 文件：

```text
docs/ai/AI工具/chatgpt.md
```

推荐内容格式：

```markdown
---
title: ChatGPT 使用记录
tags: [ai, 工具, ChatGPT]
---

# ChatGPT 使用记录

正文内容。
```

### 5.2 含图片或附件的文章

文章统一直接放在二级目录下；有图片时使用该二级目录的共享 `images/`，并给图片文件名添加文章前缀：

```text
docs/ai/AI工具/
├── chatgpt-usage.md
└── images/
    └── chatgpt-usage-screenshot.png
```

文章中使用相对路径引用图片：

```markdown
![使用截图](images/chatgpt-usage-screenshot.png)
```

这样文章会直接显示在“AI工具”下面，不会产生第三级目录。图片文件名必须包含文章前缀，避免不同文章的同名图片相互覆盖。

## 六、移动文章或调整层级

移动文章后，除了检查图片路径和其他页面链接，还要同时更新原二级目录和新二级目录的 `index.md` 内容目录。

例如，将原来位于错误位置的文章：

```text
docs/ai/example.md
```

移动到“AI工具”目录：

```text
docs/ai/AI工具/example.md
```

在 Windows PowerShell 中可以执行：

```powershell
Move-Item `
  "docs/ai/example.md" `
  "docs/ai/AI工具/example.md"
```

移动后检查以下内容：

- 文章内部的图片相对路径是否仍然正确。
- 其他 Markdown 页面是否引用了旧路径。
- 首页、分类页或其他文章中的链接是否需要更新。
- 文件名大小写是否一致。

搜索旧链接示例：

```bash
rg -n "example\.md|旧目录名" docs
```

## 七、重命名目录或文章

### 7.1 重命名目录

目录名会影响左侧栏名称和页面 URL。例如：

```text
docs/ai/AI工具/
```

改为：

```text
docs/ai/AI工具箱/
```

重命名后，目录中的 `index.md` 和文章内容通常不需要修改，但所有指向旧 URL 的链接都要检查。

### 7.2 重命名文章

例如：

```text
docs/ai/AI工具/chatgpt.md
```

改为：

```text
docs/ai/AI工具/chatgpt-使用记录.md
```

文章在左侧栏的显示名称不一定需要跟着文件名改。如果只想改显示名称，优先修改：

```yaml
---
title: ChatGPT 使用记录
---
```

如果修改了文件名，还要搜索旧文件名并更新文章正文、其他页面以及所属二级目录 `index.md` 中的链接。

## 八、删除目录或文章

删除一篇文章时，删除对应的二级目录下 `.md` 文件；如果该文章有图片，同时删除 `images/` 中带有该文章前缀的图片。

删除一个二级目录时，确认该目录中的所有文章都已处理，然后删除整个目录：

```text
docs/ai/待删除目录/
```

删除后执行严格构建，确保没有遗留链接：

```bash
.venv/Scripts/mkdocs.exe build --strict
```

## 九、不要修改 `mkdocs.yml` 中的 `nav`

当前项目依赖 MkDocs 的自动导航，因此日常新增内容不需要在 `mkdocs.yml` 中手工添加 `nav` 项。

只要把文件放进正确目录，构建时就会自动出现在左侧栏：

```text
docs/一级目录/二级目录/文章.md
```

只有在确实需要自定义排序、隐藏页面或设置不同显示名称时，才考虑改为手工配置 `nav`。一旦手工配置 `nav`，新增文件可能不会自动出现在左侧栏，维护成本也会增加。

## 十、修改后的标准验证流程

每次调整目录或文章后，按以下顺序操作：

### 10.1 检查目录结构

```bash
find docs -maxdepth 3 -type f | sort
```

确认文章没有直接放在一级目录下。

### 10.2 搜索旧链接

```bash
rg -n "旧文件名|旧目录名|旧路径" docs
```

如果搜索到旧路径，修改为新路径。

### 10.3 严格构建

Windows：

```powershell
.venv\Scripts\mkdocs.exe build --strict
```

或在 Bash 中：

```bash
.venv/Scripts/mkdocs.exe build --strict
```

构建必须成功。项目现有的 Material 版本提示和 `uslugify` 弃用提示属于警告，不是本次目录调整的构建错误。

### 10.4 本地预览

```bash
.venv/Scripts/mkdocs.exe serve
```

打开：

```text
http://127.0.0.1:8000
```

检查左侧栏：

- 一级目录是否只是分组。
- 二级目录是否出现在正确的一级目录下。
- 文章是否都位于对应二级目录下。
- `index.md` 是否作为二级目录首页显示。
- `index.md` 的“内容目录”是否包含该二级目录下全部文章，并且链接路径正确。
- 目录和页面是否出现不希望的重复名称。

### 10.5 提交并推送

```bash
git add .
git commit -m "docs: ..."
git push origin main
```

推送后，GitHub Actions 会自动构建并部署 GitHub Pages。部署成功后刷新网页；若浏览器仍显示旧导航，使用 `Ctrl+F5` 强制刷新。

## 十一、快速判断口诀

```text
一级目录：只分类，不直接放文章。
二级目录：承载一个主题，可以有 `index.md`。
index.md：目录首页和内容索引页，也是一个可点击页面。
普通 .md：具体文章，必须放在二级目录下。
文章有图片：使用“二级目录下的文章.md + 二级目录/images/文章名前缀-图片名”。
新增、删除或重命名文章：同步更新该二级目录的 `index.md` 内容目录。
修改后：检查链接，严格构建，再提交推送。
```
